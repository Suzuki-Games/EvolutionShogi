# Evolution Shogi

Unity (C#) で開発した、**成長で逆転する非対称ターン制ストラテジー**。

将棋をベースに「駒の成長（進化）」を加えたオリジナルゲーム。プレイヤーの主人公駒は最弱の「歩」として始まり、敵を倒し前進することで経験値を貯め、**分岐型の進化ツリー**を辿って最強の「勇者」を目指します。圧倒的な数的・質的不利を、自らの成長で覆していく非対称バランスがコアの体験です。

## Features

- **分岐型の進化ツリー**: 経験値で進化する際、複数の進化先から選択（例：歩→銀 or 金 → 飛車 or 角 → 勇者）。多段進化対応
- **不屈の精神（リスポーン）**: 主人公が倒されても3ターン後に「歩」として復活。何度でも這い上がれるが、成長はゼロから
- **持ち駒システム**: 取った敵駒を盤面に打ち込み可能。二歩・最奥段制限など正式ルール準拠
- **スコアベース敵AI（1ターン2回行動）**: 攻撃価値・リスク評価・位置評価で最善手を選択。1ターンに最大2回行動し、数的優位を脅威として実感させる設計
- **進化連動の脅威変化**: プレイヤーが進化するほど敵AIの攻撃モードが強化され、緊張感が動的に変化
- **脅威可視化UI**: プレイヤーターン開始時に全敵の攻撃可能マスをオレンジで表示。「ここに動いたら取られる」が一目で分かる
- **進化演出**: 画面フラッシュ＋アナウンステキスト＋スケールアニメの三段重ねでカタルシスを演出
- **WebGL対応**: GitHub Pagesでブラウザ即プレイ可能

### アーキテクチャ

```
Scripts/
├── Board/    — 盤面データ・描画・タイル管理・移動アニメーション
├── Pieces/   — 駒の継承構造（基底 Piece ＋ Hero/Ally/Enemy/King の4サブクラス）
```

ポリモーフィズムで駒種ごとの挙動（移動パターン・取られた時の処理）を共通化し、新規駒追加が容易な設計。

## 設計判断（Why this design?）

### 1. なぜ進化を「分岐型」にしたか

当初は「歩→銀→飛車→勇者」の一本道だったが、これだとプレイヤーの判断軸が「いつ進化するか」しかなかった。各ティアに2択（銀 or 金、飛車 or 角）を設けることで「**どちらを選ぶか**」という戦略的判断が加わり、毎回違う攻略パスを取れるようになった。

### 2. なぜ敵AIに「1ターン2回行動」を導入したか

1ターン1行動だと、敵17体の数的優位がほぼ機能しなかった（中央の歩しか動かない単調な展開）。2回行動に変更することで、敵駒の数が攻撃圧として実感できる設計に。同じ駒は1ターン中に1回までという制約で、操作性も維持。

### 3. なぜ「単ターン即詰み防止規則」を追加したか

2回行動を入れた直後のプレイテストで「敵が金を王の隣に打ち、同ターン2手目でその金が王を捕獲する」**絶対防御不能のコンボ**が発生した。プレイヤーが応手を挟めない瞬間に敗北するのはゲームとして破綻しているため、「2手目以降では王を取る手を禁止する」防衛規則を追加。1手目での王手は許容しつつ、応手の機会を必ず1ターン保証する設計。

### 4. なぜ「脅威可視化UI」を入れたか

AIが2回行動・包囲・進化連動などで賢くなった結果、プレイヤーが情報処理しきれなくなった。「どのマスに動いたら取られるか」を駒選択前から把握できる可視化を入れることで、プレイヤーの「考える」を支える設計に。情報過多にならないよう、駒選択時の移動ハイライトとは独立に管理している。

## My Role（AI協働での役割分担）

AIツール（Claude Code）を活用したペアプログラミングで開発しました。

### 自分の役割

- **コンセプト設計**: 「非対称な戦力差を成長で覆す」というコアテーマ設定
- **ゲームデザインの判断**: 分岐型進化・2回行動・脅威可視化など、プレイテストから得た改修方針の決定
- **プレイテストとデバッグ報告**: 単ターン即詰みコンボの発見、勇者を狙わないAIの問題発見、詰まる箇所の特定
- **Unityエディタ作業**: シーン構築・プレハブ設定・UI配置・WebGL ビルド設定

### AIの役割

- C#コードの実装・バグ修正・アーキテクチャ提案
- 設計の壁打ち相手（「この実装で副作用ないか？」のレビュー）

### 学び

AIを使いこなすにも、「**何を作りたいか**」「**何が問題か**」を正確に伝える力が必要だと実感した。AIが書くコードの正しさは「自分が要件を言語化できる範囲」に強く制約される。プレイテストで違和感を見つけ、それを言葉にして伝える――この往復こそが個人開発の核心だった。

## 前作（テトリス）からの繋がり

前作のテトリス再現では、AIに教えてもらいながら写経で作ったが、途中からコピペ中心になり「**理解が定着しない違和感**」が残った。その反省から、本作では「自分で設計判断を下す」ことを意識し、**「なぜこの仕様にしたか」を説明できる状態で進めること**を習慣にした。設計判断の詳細は [`将棋企画書.txt`](将棋企画書.txt) に全て記録している。
